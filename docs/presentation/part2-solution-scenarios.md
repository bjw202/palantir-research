## 슬라이드 13: "해결책: 데이터에 의미와 관계를 부여하다"

**레이아웃:** 좌우 분할 Before/After (50:50)
**배경:** 좌측 — 어두운 회색(#2D2D2D), 우측 — 밝은 파란색 그라디언트(#E8F4FD → #FFFFFF)
**애니메이션:** 좌측 먼저 페이드인(0.5초) → 중앙 화살표 등장 → 우측 슬라이드인(우→좌)

### 핵심 메시지
> 온톨로지는 흩어진 데이터 파편을 "살아있는 공장 지도"로 바꾸는 기술입니다.

### 콘텐츠

**좌측 — 현재: "흩어진 데이터 파편"**

```
  ┌─────┐   ┌─────┐   ┌─────┐
  │ ERP │   │ MES │   │SCADA│
  └──┬──┘   └──┬──┘   └──┬──┘
     ?          ?          ?        ← 연결 없음
  ┌─────┐   ┌─────┐   ┌─────┐
  │ QMS │   │ WMS │   │CMMS │
  └─────┘   └─────┘   └─────┘
```

- 같은 설비가 시스템마다 다른 이름
  - ERP: "EQ-001-A"
  - MES: "MACHINE_A_LINE1"
  - CMMS: "ASSET-00142"
- 불량 원인 추적: 4~5개 시스템 수동 교차 분석
- "데이터는 넘치는데, 답은 없다"

**→ 화살표 (중앙): "온톨로지 변환"**

**우측 — 목표: "연결된 객체 네트워크"**

```
        ┌──────────┐
        │  설비     │
        │ (Machine)│
        └────┬─────┘
       ┌─────┼─────┐
  ┌────▼──┐ ┌▼────┐ ┌▼─────┐
  │ 공구  │ │Lot  │ │레시피│
  │(Tool) │ │     │ │      │
  └───────┘ └──┬──┘ └──────┘
            ┌──▼──┐
            │자재  │──→ 공급사
            └─────┘
```

- 하나의 이름, 하나의 정의, 모든 팀이 공유
- 관계를 따라가면 자동으로 원인에 도달
- "데이터가 스스로 이야기한다"

### 발표자 노트
지금 우리 공장의 데이터는 왼쪽 그림처럼 7개 시스템에 흩어져 있습니다. 같은 설비인데 시스템마다 이름이 달라서 연결이 안 됩니다. 온톨로지는 이것을 오른쪽처럼 바꿉니다. 현실 세계의 설비, 제품, 자재, 작업자를 하나의 객체로 정의하고, 그 사이의 관계를 연결하는 것입니다. 마치 종이 지도를 네비게이션 앱으로 바꾸는 것과 같습니다.

## 슬라이드 14: "온톨로지란? — 30초 설명"

**레이아웃:** 4개 카드 가로 배치 (각 카드: 아이콘 + 이름 + 비유 1줄)
**배경:** 흰색(#FFFFFF), 카드별 상단 색상 바 — 파랑/초록/주황/빨강
**애니메이션:** 카드가 좌→우 순서로 0.3초 간격 플립인

### 핵심 메시지
> 온톨로지의 4가지 구성요소는 "사물, 관계, 판단, 실행"입니다.

### 콘텐츠

| | **Object (객체)** | **Link (관계)** | **Function (판단)** | **Action (실행)** |
|---|---|---|---|---|
| **아이콘** | 공장 설비 모양 | 연결선/화살표 | 계산기/두뇌 | 재생 버튼/손 |
| **색상 바** | 파랑 (#3B82F6) | 초록 (#10B981) | 주황 (#F59E0B) | 빨강 (#EF4444) |
| **한 줄 정의** | 공장의 사물과 사람 | 사물 간의 연결 | 자동 계산과 판단 | 실제 조치 실행 |
| **비유** | "이것은 CNC-14 설비이다" | "이 설비가 저 제품을 만든다" | "이 설비가 위험한지 점수를 매긴다" | "정비팀에 수리 요청을 보낸다" |
| **예시** | Machine, Lot, Tool, Recipe, Material, Operator | "가공됨", "사용됨", "공급함" | 설비 건강 점수, 불량 기여도 계산 | Lot 격리, 레시피 변경, 정비 요청 |

**하단 요약 문구:**
> "공장의 모든 사물(Object)이 관계(Link)로 연결되고, 스스로 판단(Function)하고, 필요하면 행동(Action)합니다."

### 발표자 노트
온톨로지를 어렵게 생각할 필요가 없습니다. 네 가지만 기억하시면 됩니다. 첫째, Object는 공장에 있는 모든 것 — 설비, 제품, 자재, 사람입니다. 둘째, Link는 그것들 사이의 관계 — "이 설비가 이 제품을 만든다" 같은 연결입니다. 셋째, Function은 자동 판단 — "이 설비 지금 위험한가?" 점수를 매기는 것입니다. 넷째, Action은 실제 행동 — "정비 요청을 보내라" 같은 실행입니다. 이 네 가지가 합쳐지면 데이터가 스스로 생각하고 행동하는 시스템이 됩니다.

## 슬라이드 15: "현실 세계를 디지털로 복제하다"

**레이아웃:** 좌우 분할 — 좌측 60%(공장 일러스트 + 실물), 우측 40%(디지털 표현), 화살표로 1:1 매핑
**배경:** 밝은 그레이(#F8F9FA)
**애니메이션:** 좌측 실물 항목이 하나씩 등장 → 각각에서 화살표가 우측 디지털 객체로 연결

### 핵심 메시지
> 현실 공장의 모든 것이 디지털 세계에 1:1로 대응됩니다.

### 콘텐츠

**좌측 — 현실 세계 (공장 일러스트)**

```
  🏭 공장 사진/일러스트

  ① CNC-14 설비 (실물 사진)         ──→
  ② "이 설비가 이 공정에 사용됨"     ──→
  ③ 센서 측정값 (온도 85°C)         ──→
  ④ "설비 고장 시 정비팀 호출"       ──→
  ⑤ 작업자가 Lot 검사 중            ──→
```

**우측 — 디지털 표현 (온톨로지)**

```
  ① Machine { id: "M-14", status: "active",
              healthScore: 62 }

  ② Link: Machine ─[operates]→ ProductionLine

  ③ Property: vibrationLevel = 3.8mm/s
              temperature = 85°C

  ④ Action: RequestMaintenance
            → 유효성 검사 → 권한 확인 → 실행

  ⑤ Operator ─[assignedTo]→ Lot
              certifications: ["CNC", "품질검사"]
```

**하단 강조 문구:**
> "디지털 트윈(Digital Twin)이라고도 합니다. 현실이 바뀌면 디지털도 바뀌고, 디지털의 판단이 현실의 행동으로 이어집니다."

### 발표자 노트
쉽게 말하면, 현실 공장을 컴퓨터 안에 그대로 복사해 넣는 것입니다. CNC-14라는 실제 설비는 Machine이라는 디지털 객체가 됩니다. "이 설비가 이 라인에서 돌아간다"는 사실은 Link라는 관계로 기록됩니다. 센서에서 나오는 온도, 진동 같은 값은 Property로 실시간 반영되고, "고장나면 정비팀을 부른다"는 규칙은 Action으로 자동 실행됩니다. 현실과 디지털이 1:1로 연결되어 살아 움직이는 지도가 되는 것입니다.

## 슬라이드 16: "모든 것이 연결된다 — 추적 체인"

**레이아웃:** 가로 체인 다이어그램 (7개 노드, 아이콘+텍스트, 화살표 연결), 하단에 강조 문구
**배경:** 진한 남색(#1E293B), 노드는 밝은 카드, 화살표는 하늘색 글로우
**애니메이션:** 좌측 "불량" 노드부터 순서대로 체인이 연결되며 등장 (도미노 효과)

### 핵심 메시지
> 불량이 발생하면, 관계 체인을 따라 수 초 만에 근본 원인까지 도달합니다.

### 콘텐츠

**품질 역추적 체인 (Quality Trace Chain)**

```
 [불량]     [Lot]      [설비]      [공구]      [레시피]     [자재]      [공급사]
  Defect ──→ Lot ──→ Machine ──→ Tool ──→ Recipe ──→ Material ──→ Supplier
   🔴        📦       ⚙️         🔧        📋         🧱          🏢

  "Edge    "L-0316  "CNC-14"   "T-33     "R-42     "AL-7075    "삼성
   Burr     -A"                 마모78%"   v2.3"     Lot-0892"   메탈"
   3건"
```

**하단 비교 강조:**

| | 현재 | 온톨로지 |
|---|---|---|
| 추적 방식 | 4~5개 시스템 수동 로그인, 엑셀 교차 분석 | 관계 체인 자동 탐색 |
| 소요 시간 | **5시간 ~ 3일** | **수 초 ~ 1분** |
| 결과 | 단일 원인 추정 | 복합 원인 기여도(%) 산출 |

**강조 텍스트 (큰 폰트):**
> "5시간 → 1분. 99.7% 단축."

### 발표자 노트
이 체인이 온톨로지의 핵심 가치입니다. 불량이 발생하면 시스템이 자동으로 이 연결고리를 따라갑니다. 이 Lot는 어떤 설비에서 만들었는지, 그 설비에 어떤 공구가 끼워져 있었는지, 어떤 레시피를 썼는지, 어떤 자재를 투입했는지, 그 자재는 어디서 왔는지. 현재는 이 과정에 5시간에서 길면 3일이 걸립니다. 4~5개 시스템에 각각 로그인해서 엑셀로 합쳐야 하니까요. 온톨로지에서는 이미 연결이 되어 있으니 수 초면 끝납니다.

## 슬라이드 17: "AI는 이 지도 위에서 일한다"

**레이아웃:** 3층 스택 다이어그램 (아래→위), 각 층에 아이콘과 설명, 우측에 비유 텍스트
**배경:** 흰색, 각 층에 색상 그라디언트 — 1층 회색, 2층 파랑, 3층 보라
**애니메이션:** 1층부터 아래에서 위로 쌓이며 등장 (빌딩 블록 효과)

### 핵심 메시지
> 온톨로지 = 지도, AI = 내비게이션. 지도 없는 내비는 길을 잃습니다.

### 콘텐츠

```
  ┌─────────────────────────────────────────────────────┐
  │  3층: AI 에이전트 — "지도 위의 내비게이터"              │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
  │  │ 품질 진단 │  │ 예지정비  │  │ 생산 최적화│           │
  │  │  Agent   │  │  Agent   │  │  Agent   │           │
  │  └──────────┘  └──────────┘  └──────────┘           │
  │  "불량 왜 났지?"  "이 설비 언제 고장?" "납기 맞출 수 있나?"│
  ├─────────────────────────────────────────────────────┤
  │  2층: 온톨로지 — "의미의 지도"                          │
  │  15개 객체 + 14개 관계 + 7개 판단 + 7개 실행             │
  │  → 모든 설비, 제품, 자재, 작업자가 연결된 살아있는 지도    │
  ├─────────────────────────────────────────────────────┤
  │  1층: 데이터 통합 — "원재료"                            │
  │  ERP + MES + SCADA + QMS + WMS + CMMS + PLM          │
  │  → 7개 시스템의 데이터를 하나로 모으는 기반               │
  └─────────────────────────────────────────────────────┘
```

**우측 비유 (큰 텍스트):**

| 층 | 비유 |
|---|---|
| 3층 AI | 내비게이션 앱 (길 안내) |
| 2층 온톨로지 | 정밀 지도 (도로, 건물, 교통) |
| 1층 데이터 | 위성 사진 (날것 그대로) |

> "위성 사진(데이터)만으로는 길을 찾을 수 없습니다.
> 지도(온톨로지)로 정리해야 하고,
> 내비(AI)가 있어야 최적 경로를 알려줍니다."

### 발표자 노트
많은 분이 AI를 도입하면 모든 문제가 해결된다고 생각하십니다. 하지만 AI는 내비게이션과 같습니다. 아무리 좋은 내비라도 지도가 없으면 길을 찾을 수 없습니다. 1층은 7개 시스템에서 데이터를 모으는 것, 위성 사진을 찍는 단계입니다. 2층 온톨로지는 그 데이터에 의미와 관계를 부여해서 정밀 지도로 만드는 단계입니다. 그리고 3층에서 AI 에이전트가 이 지도 위에서 "불량 원인은 무엇인가", "이 설비는 언제 고장 나는가", "납기를 맞출 수 있는가"라는 질문에 답하는 것입니다.

## 슬라이드 18: "AI에게 초능력을 주지 않는다"

**레이아웃:** 중앙에 AI 에이전트 아이콘(원형), 주변 3개 방패 아이콘(삼각 배치), 각 방패에 안전장치 설명
**배경:** 흰색, 중앙 AI 아이콘 주변에 파란 보호막 시각 효과
**애니메이션:** 중앙 AI 등장 → 3개 방패가 동시에 펼쳐지며 보호막 형성

### 핵심 메시지
> AI는 "제안"만 합니다. 최종 결정과 실행은 반드시 사람이 합니다.

### 콘텐츠

**중앙: AI 에이전트**
```
       "제안합니다"
         ┌───┐
         │ AI│
         └───┘
```

**방패 1 (상단) — 인간 승인 필수**
```
  🛡️ 승인 게이트
  ─────────────
  AI: "레시피를 롤백하는 것이 좋겠습니다"
  → 공정 엔지니어 검토 → 품질팀 확인 → 생산팀장 승인
  → 그제서야 실행

  ✕ AI가 혼자 설비를 멈추거나 레시피를 바꿀 수 없음
```

**방패 2 (좌하단) — 유효성 검사 + 감사 로그**
```
  🛡️ 감사 추적
  ─────────────
  모든 AI 제안과 실행에 기록이 남음:
  • 누가 요청했는가
  • 어떤 근거로 제안했는가
  • 누가 승인했는가
  • 결과는 어떠했는가

  → IATF 16949 / ISO 감사 대응 가능
```

**방패 3 (우하단) — 접근 범위 제한**
```
  🛡️ 권한 통제
  ─────────────
  AI도 사람과 동일한 보안 정책 적용:
  • 볼 수 있는 데이터 제한 (행·열 수준)
  • 실행 가능한 Action 제한 (역할별)
  • 공장 A의 AI는 공장 B 데이터 접근 불가

  → AI에게 "열쇠 꾸러미"를 주지 않음
```

### 발표자 노트

## 슬라이드 21: "시나리오 A — AI가 원인을 찾는 과정"

**레이아웃:** 5단계 가로 플로우차트, 각 단계에 아이콘 + 짧은 설명, 단계별 화살표 연결

### 핵심 메시지
> 불량 보고에서 정상 복귀까지, AI가 관계 그래프를 따라 5단계로 문제를 해결합니다.

### 콘텐츠

```

  ┌──────────┐     ┌──────────┐      ┌──────────┐      ┌──────────┐      ┌──────────┐
  │  📋 보고  │ ──→ │  🔍 탐색  │ ──→  │  📊 분석  │ ──→  │  💡 제안  │ ──→  │  ✅ 실행  │
  └──────────┘     └──────────┘      └──────────┘      └──────────┘      └──────────┘
       │                │                  │                  │                  │

  자동 알림 발생    Recipe → Material   공구 마모 28%      + 공구 교체 예약      감사 로그 기록
                   → Supplier         정비 런인 12%       공정+품질+생산       학습 데이터 축적
                   (3초 소요)          자재 5%, 작업자 3%   3단계 승인 필요      "이 패턴 기억"
```

### 발표자 노트
AI가 원인을 찾는 과정을 5단계로 보여드립니다. 첫째, 불량이 보고되면 자동으로 알림이 발생합니다. 둘째, 시스템이 관계 체인을 따라 Lot에서 설비, 공구, 레시피, 자재, 공급사까지 3초 만에 탐색합니다. 셋째, 각 요인이 불량에 얼마나 기여했는지 퍼센트로 계산합니다. 넷째, 구체적인 조치를 제안합니다. 다섯째, 담당자 승인 후 자동 실행되고, 이 경험이 학습 데이터로 축적되어 다음번에는 같은 조건이 발생하면 사전에 경고합니다.

이것은 제조 현장에서 거의 매주 벌어지는 상황입니다. 정비팀은 "지금 안 하면 고장 난다"고 하고, 생산팀은 "지금은 안 된다, 긴급 주문이다"고 합니다. 결국 "다음 주"로 미루고, 수요일에 정말 고장이 나서 12시간 다운타임에 1억 원 이상의 손실이 발생합니다. AI는 이 갈등에 데이터를 가져옵니다. 건강 점수가 하루에 1.5점씩 떨어지고 있으니 금요일까지 미루면 고장 확률이 23%라고 알려줍니다. 그리고 "오늘 교대 시간에 4시간만 하면 되고, 대체 설비로 납기도 맞출 수 있다"는 구체적인 대안을 제시합니다. 감이 아니라 데이터로 결정하는 것입니다.

## 슬라이드 24: "세 시나리오의 시너지 — 닫힌 루프"

**레이아웃:** 순환 다이어그램 (3개 노드 원형 배치 + 화살표), 중앙에 "온톨로지" 허브, 하단에 연간 절감 효과
**배경:** 흰색, 순환 다이어그램은 그라디언트 화살표(파랑→보라→초록→파랑)
**애니메이션:** 중앙 "온톨로지" 허브 등장 → 3개 노드가 120도 간격으로 펼쳐짐 → 화살표가 순환 방향으로 그려짐 → 하단 절감 효과 펀치인

### 핵심 메시지
> 세 시나리오는 동일한 온톨로지 위에서 돌아가며, 서로가 서로를 똑똑하게 만듭니다.

### 콘텐츠

**순환 다이어그램:**

```
                    A. 품질 진단
                   ┌──────────┐
                   │ 불량 패턴 │
                   │ 데이터 축적│
                   └─────┬────┘
                 ↗       │       ↘
    C→A: 정비 후         │        A→B: 불량 패턴이
    품질 변화 학습        │        이상 감지 정확도 향상
               │         │         │
   ┌───────────┴─┐  ┌────▼────┐  ┌─▼──────────┐
   │ C. 예지정비  │  │ 온톨로지 │  │ B. 이상 대응 │
   │ 최적 시점    │  │  (허브)  │  │ 실시간 감지  │
   │ 예측        │  └─────────┘  │ + 자동 대응  │
   └──────────┬──┘               └──┬──────────┘
               ↖                   ↙
                 B→C: 이상 빈도가
                 정비 예측 모델 개선
```

**시너지 설명 (3개 화살표):**

| 흐름 | 의미 |
|---|---|
| A → B | 불량 패턴("Edge Burr 전 진동 패턴") → 이상 감지 모델이 더 정확해짐 |
| B → C | 이상 발생 빈도 증가 → 설비 열화 가속 판단 → 정비 시점 더 정확해짐 |
| C → A | 정비 전후 품질 변화 데이터 → 불량 기여도 분석이 더 정확해짐 |

**중앙 "온톨로지" 허브 설명:**
> 세 에이전트가 **같은 Machine, 같은 Lot, 같은 Recipe 객체**를 공유하기 때문에 시너지가 가능합니다.

**하단 — 연간 절감 효과 (대형 폰트, 강조 박스):**

### 발표자 노트
마지막으로 가장 중요한 포인트입니다. 이 세 시나리오는 따로 도는 것이 아닙니다. 시나리오 A에서 축적된 불량 패턴 데이터가 시나리오 B의 이상 감지를 더 정확하게 만들고, 시나리오 B에서 쌓인 이상 발생 데이터가 시나리오 C의 정비 예측을 더 정확하게 만들고, 시나리오 C에서 정비 전후의 품질 변화가 시나리오 A의 분석을 더 정확하게 만듭니다. 이것이 가능한 이유는 세 에이전트가 동일한 온톨로지 — 같은 설비, 같은 Lot, 같은 레시피 객체를 공유하기 때문입니다. 종합하면 연간 108억에서 188억원의 절감이 기대되며, 투자 회수 기간은 6개월에서 12개월입니다.
