## 1. 데이터 사일로의 실체

### 1-1. 스마트팩토리의 전형적 시스템 구성도

제조 현장은 단일 시스템이 아니다. 수십 년에 걸쳐 도입된 이기종 시스템들이 각자의 영역을 지키고 있다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    경영/계획 계층 (Level 4)                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                         │
│  │   ERP   │  │   SCM   │  │   PLM   │                         │
│  └────┬────┘  └────┬────┘  └────┬────┘                         │
├───────┼────────────┼────────────┼───────────────────────────────┤
│       │   실행/운영 계층 (Level 3)                                │
│  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐  ┌─────────┐           │
│  │   MES   │  │   QMS   │  │   WMS   │  │  CMMS   │           │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘           │
├───────┼────────────┼────────────┼────────────┼──────────────────┤
│       │      제어 계층 (Level 2/1/0)                              │
│  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐           │
│  │  SCADA  │  │   PLC   │  │  센서/  │  │  DCS    │           │
│  │         │  │         │  │ 액추에이터│  │         │           │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘           │
└─────────────────────────────────────────────────────────────────┘
```

### 1-2. 각 시스템이 소유하는 데이터

| 시스템 | 주요 데이터 | 데이터 특성 |
|--------|-------------|-------------|
| **ERP** (SAP, Oracle 등) | 생산 계획, BOM, 원가, 구매/발주, 재무 | 트랜잭션 기반, 배치 갱신, 일/주 단위 |
| **MES** (Manufacturing Execution System) | 작업 지시, Lot 추적, 공정 이력, 설비 가동 상태, 작업자 투입 이력 | 준실시간, 이벤트 기반, 초/분 단위 |
| **SCADA/PLC** (Supervisory Control and Data Acquisition) | 설비 센서값(온도, 압력, 진동, 전류), 알람 이력, 제어 명령 로그 | 실시간 스트리밍, 밀리초 단위, 대용량 시계열 |
| **QMS** (Quality Management System) | 검사 결과, 불량 유형 분류, CAPA(시정/예방 조치), SPC 데이터 | 이벤트 기반, 수동 입력 비중 높음 |
| **WMS** (Warehouse Management System) | 원자재/완제품 재고, 입출고 이력, 로케이션, 유통기한 | 트랜잭션 기반, 바코드/RFID 연동 |
| **CMMS** (Computerized Maintenance Management System) | 설비 정비 이력, 예방 정비 스케줄, 부품 재고, 정비 인력 배정 | 이벤트/스케줄 기반, 수동 입력 혼재 |
| **PLM** (Product Lifecycle Management) | 설계 도면, 공정 사양(Recipe/Spec), 변경 이력(ECO/ECN), 시뮬레이션 결과 | 문서 중심, 버전 관리, 비정형 데이터 포함 |

### 1-3. 시스템 간 데이터 단절의 구체적 지점과 원인

#### 기술적 원인

**프로토콜 차이:**
- SCADA/PLC 계층은 OPC-UA, Modbus, PROFINET 등 산업용 프로토콜을 사용한다.
- MES/ERP 계층은 REST API, SOAP, RFC(SAP), JDBC/ODBC를 사용한다.
- 이 두 계층 사이에 프로토콜 변환(Gateway)이 필요하며, 변환 과정에서 데이터 손실과 지연이 발생한다.

**마스터 데이터 불일치:**
- ERP의 설비 코드: `EQ-001-A`
- MES의 설비 코드: `MACHINE_A_LINE1`
- CMMS의 설비 코드: `ASSET-00142`
- 같은 설비를 가리키는 세 개의 서로 다른 ID. 이를 매핑하는 마스터 데이터 관리(MDM)가 없으면 시스템 간 연결 자체가 불가능하다.

**실시간 vs. 배치 갭:**
- 센서 데이터는 밀리초 단위로 생성되지만, ERP는 일 단위 배치로 데이터를 처리한다.
- MES에서 불량이 발생해도 ERP의 생산 실적에 반영되기까지 수 시간~하루가 걸린다.
- 이 시간 갭 동안 경영층은 "어제의 데이터"로 오늘의 결정을 내린다.

#### 조직적 원인

**부서별 소유권:**
- 생산팀이 MES를, 품질팀이 QMS를, 설비팀이 CMMS를, IT팀이 ERP를 각각 "소유"한다.
- 각 부서는 자기 시스템의 데이터 정의, 입력 규칙, 접근 권한을 독립적으로 관리한다.
- 부서 간 데이터 공유를 위해서는 "요청서"를 쓰고 IT에 개발을 의뢰해야 한다. 이 과정이 수 주~수 개월이 걸린다.

**벤더 종속(Vendor Lock-in):**
- ERP는 SAP, MES는 Rockwell, QMS는 ETQ, CMMS는 Maximo 등 서로 다른 벤더 제품이다.
- 각 벤더는 자사 에코시스템 내 통합은 쉽게 하지만, 타사 시스템과의 연동은 커스텀 개발을 요구한다.
- 벤더 업그레이드 시 기존 인터페이스가 깨지는 일이 빈번하다.

**도입 시기 차이:**
- ERP: 2005년 도입, 온프레미스
- MES: 2015년 도입, 온프레미스
- IoT 플랫폼: 2022년 도입, 클라우드
- 10~20년에 걸쳐 도입된 시스템들이 설계 철학 자체가 다르다. 2005년에는 "실시간 데이터 연동"이라는 개념 자체가 요구사항에 없었다.

### 1-4. 데이터 사일로가 만드는 구체적 업무 장애

**사례 1: 불량 원인 추적에 3일이 걸리는 현장**
고객 클레임이 접수되면, 품질팀은 QMS에서 불량 이력을 확인하고, MES에서 해당 Lot의 공정 이력을 조회하고, SCADA에서 해당 시점의 센서 데이터를 뽑고, CMMS에서 설비 정비 이력을 확인해야 한다. 4개 시스템에 각각 로그인하고, 시간 범위를 맞추고, 엑셀로 데이터를 병합하는 데 최소 2~3일이 소요된다. 그 사이에도 동일 불량이 계속 생산될 수 있다.

**사례 2: 재고-생산 불일치로 라인 정지**
WMS의 원자재 재고와 ERP의 생산 계획이 실시간 동기화되지 않아, 생산 라인이 시작된 후에야 특정 자재가 부족하다는 것을 알게 된다. 라인 정지 비용은 산업에 따라 시간당 $10,000~$260,000에 달한다([Siemens, 2024](https://blog.siemens.com/2024/07/the-true-cost-of-an-hours-downtime-an-industry-analysis/)).

**사례 3: 설비 정비 이력과 품질 데이터 미연결**
CMMS에 기록된 정비 이력(베어링 교체, 캘리브레이션)과 QMS의 품질 데이터가 연결되지 않아, "정비 후 품질이 개선되었는가?"를 정량적으로 평가할 수 없다. 정비팀은 경험에 의존하고, 품질팀은 통계에 의존하지만, 둘은 서로의 데이터를 보지 못한다.

**사례 4: 공급사 품질 문제의 늦은 발견**
입고 검사 데이터(QMS)와 해당 자재를 사용한 공정 결과(MES)가 연결되지 않아, 특정 공급사 Lot에서 품질 이슈가 있었는지 사후에야 파악된다. 문제 자재가 이미 수백 개 제품에 투입된 후 회수 비용이 기하급수적으로 증가한다.

**사례 5: 에너지 비용 최적화 불가**
SCADA의 설비 전력 소비 데이터와 ERP의 생산 스케줄이 연결되지 않아, "어떤 제품을 어떤 시간대에 생산하면 에너지 비용이 최소화되는가?"라는 질문에 답할 수 없다.

### 1-5. 기존 해결 시도가 근본적으로 부족한 이유

**ETL (Extract, Transform, Load):**
- 데이터를 뽑아서, 변환하고, 적재한다. 하지만 이것은 "데이터의 복사"이지 "데이터의 연결"이 아니다.
- 원천 시스템이 변경되면 ETL 파이프라인이 깨진다. 중대형 공장에서 ETL 파이프라인 수는 수백~수천 개에 달하며, 이를 유지보수하는 것 자체가 하나의 프로젝트가 된다.
- ETL로 이동된 데이터는 이동 시점의 스냅샷이다. 실시간성이 없다.

**데이터 레이크 (Data Lake):**
- 모든 데이터를 한 곳에 모은다. 그러나 Gartner는 빅데이터 프로젝트의 실패율이 85%에 달한다고 지적하며, 데이터 레이크의 "Data Swamp화" 위험을 경고해왔다([Gartner, 2014](https://www.gartner.com/en/newsroom/press-releases/2014-07-28-gartner-says-beware-of-the-data-lake-fallacy)).
- 근본 원인: 데이터는 모였지만 **의미(semantic meaning)와 관계(relationship)가 없다**. MES의 `MACHINE_A_LINE1`과 CMMS의 `ASSET-00142`가 같은 설비라는 것을 데이터 레이크는 모른다.
- 반도체 업계 사례에서도, 데이터 레이크에 데이터를 넣었지만 "spatial, time, hierarchical relationship" 없이는 어떤 고급 분석도 불가능하다고 지적한다([Semi Engineering](https://semiengineering.com/making-the-most-of-data-lakes/)).

**미들웨어 / ESB (Enterprise Service Bus):**
- 시스템 간 메시지를 중계하는 버스를 놓는다. 하지만 이것은 "연결 파이프"이지 "의미의 통일"이 아니다.
- ESB는 A 시스템의 메시지를 B 시스템이 이해할 수 있는 포맷으로 변환하지만, 비즈니스 의미까지 변환하지는 않는다.
- N개 시스템을 연결하면 이론적으로 N(N-1)/2개의 인터페이스가 필요하다. 시스템이 7개면 21개의 인터페이스. 이것이 "인터페이스 지옥(Integration Hell)"이다.

> **핵심 진단:** 기존 접근은 모두 "데이터를 이동"하는 데 집중했다. 하지만 문제의 본질은 이동이 아니라 **의미의 통일과 관계의 구축**이다. 2023년 산업 조사에 따르면, 76%의 기업이 데이터 사일로가 부서 간 교류를 방해한다고 인정했으며, 74%는 이를 경쟁 열위로 인식하고 있다([XPLM, 2023](https://www.xplm.com/news/press/industry-study-2023-companies-cannot-control-their-data-silos/)).

## 2. 품질 추적 불가의 구조적 원인

### 2-1. 제조 품질 추적의 이상적 경로

제품 품질 문제가 발생했을 때, 엔지니어가 원인을 찾기 위해 추적해야 하는 이상적 경로는 다음과 같다.

```
Lot (생산 단위)
 ├── processedOn → Machine (어떤 설비에서 가공했는가)
 │                    ├── mountedWith → Tool (어떤 공구/지그를 사용했는가)
 │                    ├── usedRecipe → Recipe (어떤 공정 조건을 적용했는가)
 │                    └── takenFrom ← Measurement (온도, 습도, 진동, 전류)
 ├── usedMaterial → Material (어떤 원자재/부자재를 투입했는가)
 │                    └── provides ← Supplier (어떤 공급사에서 온 Lot인가)
 └── processedOn → Machine → Shift ← assignedTo ← Operator (어떤 작업자가 담당했는가)
```

이 추적 경로가 완전하면, "이 불량은 특정 공급사의 자재 Lot + 특정 공구의 마모 + 야간조의 셋업 오류가 복합 작용한 결과"라는 근본 원인 분석(Root Cause Analysis)이 가능하다.

### 2-2. 현실에서 추적 경로가 끊기는 지점들

**MES: Lot-Machine 연결은 있지만, Tool 정보가 없다**
- MES는 "Lot L-2024-0317이 Machine M-14에서 14:30~15:15에 가공됨"을 기록한다.
- 하지만 그 시점에 M-14에 장착된 Tool이 무엇인지는 MES가 관리하지 않는다. Tool 관리는 별도의 Tool Management System(TMS)이나 CMMS에 있다.
- **끊김 지점:** MES의 `machineId`와 TMS의 `toolId`를 특정 시간대에 매핑하는 로직이 없다.

**QMS: 불량 데이터는 있지만, 공정 조건 연결이 약하다**
- QMS에는 "Lot L-2024-0317에서 Edge Burr 불량 3건 발생"이 기록된다.
- 하지만 해당 불량 발생 시점의 공정 조건(온도, 압력, 피드 속도)이 QMS에 직접 기록되지 않는다. 공정 조건은 SCADA/PLC에 있다.
- **끊김 지점:** QMS의 `defectId`와 SCADA의 센서 데이터를 시간 동기화하여 연결하는 메커니즘이 없다.

**센서 데이터: 존재하지만 Lot 매핑이 안 된다**
- SCADA는 "M-14의 진동 값이 14:42에 4.2mm/s로 급등"을 기록한다.
- 하지만 14:42에 M-14에서 어떤 Lot가 가공 중이었는지를 SCADA는 모른다. Lot 정보는 MES에 있다.
- **끊김 지점:** SCADA의 타임스탬프와 MES의 Lot-Machine 시간 매핑을 실시간으로 결합하는 통합 계층이 없다.

**자재 추적: 입고와 투입 사이의 단절**
- WMS는 "공급사 S-07의 자재 Lot M-2024-0892가 3월 10일에 입고됨"을 기록한다.
- MES는 "Lot L-2024-0317에 자재가 투입됨"을 기록하지만, WMS의 자재 Lot ID와 연결되지 않는 경우가 많다.
- 특히 벌크 자재(액체, 분말 등)는 탱크에서 혼합되면서 개별 Lot 추적이 물리적으로 불가능해진다.

**작업자-공정 연결: 형식적 기록**
- MES에 작업자 ID가 기록되지만, 실제로 해당 작업자가 셋업을 했는지, 가공 중 개입했는지, 교대 시점에 인수인계가 제대로 되었는지는 알 수 없다.
- 작업자별 품질 편차 분석은 "누가 담당할 때 불량이 많았는가"를 보는 수준이며, "무엇을 다르게 했는가"까지 추적하지 못한다.

### 2-3. 수작업 추적의 한계

현장에서 품질 문제가 발생하면 엔지니어는 다음과 같은 수작업 추적을 수행한다.

```
1. QMS에서 불량 이력 조회 → 엑셀로 내보내기           (30분)
2. MES에서 해당 Lot의 공정 이력 조회 → 엑셀로 병합     (1시간)
3. SCADA에서 해당 시간대 센서 데이터 추출 → 엑셀로 병합  (1~2시간)
4. CMMS에서 해당 설비의 최근 정비 이력 조회              (30분)
5. WMS에서 투입 자재 Lot 추적 (가능한 경우)              (30분~1시간)
6. 데이터 정합성 확인 및 시간축 정렬                      (1~2시간)
7. 원인 가설 수립 및 검증                                 (2~4시간)
─────────────────────────────────────────────────────
총 소요 시간: 최소 6시간 ~ 최대 3일 (복잡한 경우)
```

이 과정의 근본적 문제:
- **시간 소요:** 분석 시간 동안 동일 조건에서 불량이 계속 생산될 수 있다.
- **인적 오류:** 4~5개 시스템의 데이터를 수동으로 병합할 때 시간대 불일치, 코드 매핑 실수가 발생한다.
- **사후 분석만 가능:** 불량이 발생한 후에야 분석을 시작하므로, 예방적 품질 관리가 구조적으로 불가능하다.
- **암묵지 의존:** 숙련 엔지니어의 경험에 의존하여 "어디를 봐야 하는지"를 판단한다. 해당 인력이 퇴직하면 추적 능력 자체가 사라진다.

### 2-4. 품질 비용(Cost of Quality) 관점의 손실 규모

품질 비용(CoQ)은 네 가지로 구성된다.

| CoQ 구성 요소 | 설명 | 추적 불가 시 영향 |
|---------------|------|-------------------|
| **예방 비용** (Prevention Cost) | 교육, 공정 설계, SPC | 예방 활동의 효과를 측정할 수 없어 예산 정당화 실패 |
| **평가 비용** (Appraisal Cost) | 검사, 테스트, 감사 | 어디를 집중 검사해야 하는지 모르므로 전수 검사로 회귀 |
| **내부 실패 비용** (Internal Failure) | 재작업, 폐기, 재검사 | 원인 파악 지연으로 재작업 범위 확대 |
| **외부 실패 비용** (External Failure) | 반품, 리콜, 보상, 신뢰 손실 | 문제 자재/공정 추적 실패로 리콜 범위 과대 산정 |

**산업 평균 데이터:**
- 제조 기업의 품질 비용(Cost of Poor Quality)은 매출의 **15~20%** 에 달한다. 제품 복잡도에 따라 5%에서 35%까지 편차가 있다([IISE](https://www.iise.org/details.aspx?id=22118), [Parsec](https://www.parsec-corp.com/blog/cost-of-quality)).
- 연 매출 1,000억 원 규모의 제조사라면, 품질 비용이 연간 **150~200억 원** 에 달한다는 의미다.
- 이 중 상당 부분이 "추적 불가로 인한 과잉 대응 비용"이다. 원인을 특정할 수 없으니 의심되는 모든 Lot를 폐기하고, 의심되는 모든 설비를 정지시키는 것이다.

> **핵심 진단:** 품질 추적 불가의 본질은 "데이터가 없는 것"이 아니라, **데이터 간 관계(Lot→Machine→Tool→Recipe→Material)가 시스템적으로 구축되지 않은 것**이다. 개별 시스템은 각자의 데이터를 충실히 기록하고 있지만, 그것들을 하나의 추적 그래프로 연결하는 층이 없다.

## 3. 의사결정 병목 맵

### 3-1. 이상 발생 → 조치까지의 타임라인 상세 분석

설비에서 이상이 발생한 후 최종 조치까지의 전형적 경로를 단계별로 분석한다.

```
[이상 발생] → [인지] → [정보 수집] → [원인 분석] → [조치 결정] → [실행]
   T=0       T+?분     T+?시간       T+?시간       T+?시간      T+?시간
```

| 단계 | 소요 시간 (현실) | 병목 원인 |
|------|-----------------|-----------|
| **① 이상 발생 → 인지** | 5분 ~ 2시간 | 센서 임계치 설정이 부정확하거나, 알람이 너무 많아 핵심 알람이 묻힘 (알람 피로, Alarm Fatigue). SCADA 모니터를 상시 감시하는 인력 부재. 야간/주말에는 인지 자체가 크게 지연됨. |
| **② 인지 → 정보 수집** | 30분 ~ 4시간 | 이상을 인지한 후 "무슨 일인가?"를 파악하기 위해 MES, SCADA, QMS, CMMS에 각각 접속하여 데이터를 수집해야 함. 시스템별 로그인, 데이터 포맷 차이, 시간축 정렬 등에 시간 소요. |
| **③ 정보 수집 → 원인 분석** | 1시간 ~ 1일 | 수집한 데이터를 종합하여 원인 가설을 세우고 검증. 숙련 엔지니어의 경험에 크게 의존. 데이터가 불완전하면 현장 확인이 추가로 필요. |
| **④ 원인 분석 → 조치 결정** | 30분 ~ 4시간 | 원인이 파악되어도 "어떻게 조치할 것인가"의 결정에 여러 부서(생산, 품질, 설비, 공정) 간 협의가 필요. 의사결정권자가 부재 시 추가 지연. |
| **⑤ 조치 결정 → 실행** | 15분 ~ 2시간 | 결정된 조치를 현장에 전달하고 실행하는 단계. 작업 지시서 작성, 자재/부품 확보, 인력 배치 등이 필요. |

**합산:** 최소 2시간 ~ 최대 3일+

Siemens의 2024년 보고서에 따르면, 세계 500대 기업의 비계획 다운타임이 연간 매출의 11%를 잠식하며, 총 **1.4조 달러** 규모다. 이는 2019~2020년의 8,640억 달러(매출 대비 8%)에서 크게 증가한 수치다([Siemens, 2024](https://assets.new.siemens.com/siemens/assets/api/uuid:1b43afb5-2d07-47f7-9eb7-893fe7d0bc59/TCOD-2024_original.pdf)).

### 3-2. 구체적 시나리오: "설비 진동 이상 감지 후 조치까지"

#### 현재 프로세스 (As-Is)

```
14:42  M-14 설비 진동 4.2mm/s 감지 (정상 범위: ~3.5mm/s)
       → SCADA 알람 발생, 모니터에 경고등 점등

14:55  교대조 운전원이 알람 확인 (다른 알람 처리 중이었음)
       → "진동 이상? 잠깐 보자..." 다른 센서값(온도, 전류) 확인
       → 온도는 정상 범위, 전류 약간 높음
       → "일단 지켜보자" 판단 (현장에서 가장 흔한 의사결정)

15:30  진동 4.8mm/s로 추가 상승
       → 운전원이 설비팀 담당자에게 전화
       → 담당자: "지금 다른 라인에 있어서 30분 후에 갈게"

16:00  설비팀 담당자 도착, 현장 확인
       → CMMS에서 정비 이력 조회: "지난주 베어링 교체했는데..."
       → MES에서 현재 가동 중인 Lot/Recipe 확인
       → "혹시 레시피 바뀐 거 있나?" → 공정팀에 문의

16:30  공정팀 확인: "어제 레시피 변경 있었음 (피드 속도 15% 증가)"
       → 설비팀: "그거 때문일 수 있다. 원래 레시피로 돌려보자"
       → 생산팀장에게 보고, 라인 정지 승인 요청

16:45  생산팀장 승인 (생산 계획 영향 검토 후)
       → 레시피 롤백 + 설비 점검 시작

17:30  설비 재가동, 진동 정상화 확인
       → 약 2시간 45분 동안의 품질 리스크 노출 + 약 45분 라인 정지
```

#### 이상적 프로세스 (온톨로지 + AI 기반 To-Be)

```
14:42  M-14 설비 진동 4.2mm/s 감지
       → 시스템이 자동으로 관련 컨텍스트 조합:
         - Machine M-14의 최근 정비 이력 (CMMS)
         - 현재 가동 중인 Lot/Recipe (MES)
         - 레시피 변경 이력 (PLM → MES)
         - 해당 Tool의 누적 사용 시간 (TMS)
         - 동일 레시피 적용 시 타 설비의 진동 패턴 비교

14:43  AI 분석 완료:
       "M-14 진동 이상의 추정 원인:
        1순위 (확률 78%): 어제 적용된 레시피 변경(피드 속도 +15%)과
                         지난주 교체된 베어링의 런인(run-in) 기간 중복
        2순위 (확률 15%): Tool T-33 마모 (누적 사용 847시간/권장 교체 1000시간)

        권장 조치:
        [즉시] 피드 속도를 기존 레시피로 롤백 → Action: Recipe Rollback
        [24시간 내] Tool T-33 교체 스케줄링 → Action: Maintenance Request
        [모니터링] 베어링 런인 완료까지 진동 임계치 일시 상향 (4.0mm/s → 4.5mm/s)"

14:44  담당 엔지니어에게 알림 + 조치 승인 요청 (모바일 푸시)

14:46  엔지니어 승인: "레시피 롤백 승인, Tool 교체는 야간 정비 시 실행"
       → Action 자동 실행: MES에 레시피 변경 명령, CMMS에 정비 요청 생성

14:48  레시피 롤백 완료, 진동 안정화 추세 확인
       → 라인 정지 없이 조치 완료. 총 소요 시간: 6분
```

**차이:** 2시간 45분 → 6분. 라인 정지 45분 → 0분.

### 3-3. 수동 에스컬레이션의 문제점

**정보 수집 지연:**
에스컬레이션을 받은 상위 관리자는 "무슨 일인데?"라고 묻고, 보고자는 다시 데이터를 정리해서 설명해야 한다. 전화로 전달되는 정보는 불완전하고, 이메일로 보내는 엑셀 첨부파일은 최신 상태가 아닐 수 있다.

**담당자 확인 지연:**
"이건 누가 담당이지?" → 조직도 확인 → 전화 → 부재중 → 다른 사람에게 전달. 특히 여러 부서가 관련된 문제(공정+설비+품질)에서 "누가 의사결정권을 가지는가"가 불명확한 경우가 많다.

**판단 근거 부족:**
의사결정자에게 전달되는 정보가 "진동 이상 발생"이라는 한 줄 요약뿐이면, "라인을 멈춰야 하나 말아야 하나?"를 판단할 근거가 없다. 결국 보수적 판단(라인 정지)을 내리거나, 낙관적 판단(계속 가동)을 내리게 되며, 어느 쪽이든 최적이 아니다.

### 3-4. 야간/주말 상황에서의 의사결정 공백

제조 현장의 많은 공장은 24시간 가동(3교대)이지만, 의사결정 구조는 주간 근무 기준으로 설계되어 있다.

```
주간 (08:00~17:00):  공정 엔지니어 3명, 품질 엔지니어 2명, 설비 엔지니어 2명
                      → 문제 발생 시 즉시 다부서 협의 가능

야간 (22:00~06:00):  교대 운전원 2명, 당직 설비 기사 1명
                      → 복합적 문제 발생 시 전화 에스컬레이션 필요
                      → 전문 엔지니어는 수면 중 → 응답 지연 30분~1시간
                      → 원격 접속 환경 미비 시 출근 필요 → 추가 1~2시간

주말/공휴일:         야간과 유사하지만, 자재/부품 확보 불가 추가
                      → "부품이 월요일에 온다"는 이유로 2일간 감속 운전
```

> **핵심 진단:** 의사결정 병목의 본질은 "사람이 느린 것"이 아니라, **필요한 정보가 흩어져 있고, 판단 로직이 시스템화되지 않았으며, 실행 경로가 수동인 것**이다. 사람은 정보를 모으고 판단하는 데는 뛰어나지만, 4~5개 시스템에서 데이터를 수집하여 교차 분석하는 것은 사람이 할 일이 아니다.

## 4. 기존 접근이 실패하는 이유

### 4-1. BI 대시보드의 한계

BI 대시보드(Tableau, Power BI, Grafana 등)는 스마트팩토리에서 가장 많이 도입된 분석 도구다. 하지만 현장에서 듣는 불만은 한결같다.

**"현황은 보이는데, 왜 그런지 모르겠고, 뭘 해야 하는지도 모르겠다."**

```
BI 대시보드가 보여주는 것:
  ✓ "현재 불량률 3.2%"
  ✓ "설비 M-14 가동률 82%"
  ✓ "이번 주 에너지 소비량 전주 대비 +7%"

BI 대시보드가 답하지 못하는 것:
  ✗ "불량률이 왜 3.2%인가?" (원인 진단)
  ✗ "M-14를 지금 멈춰야 하나?" (의사결정 지원)
  ✗ "에너지 소비를 줄이려면 어떤 스케줄로 바꿔야 하나?" (처방)
  ✗ "이 조치를 실행해라" (실행)
```

BI 대시보드는 **"What"에는 답하지만 "Why"와 "What to do"에는 답하지 못한다.** 현장 엔지니어가 대시보드를 보고 다시 엑셀을 여는 이유다.

### 4-2. 단순 ML 모델의 한계

예측 정비(Predictive Maintenance) ML 모델은 제조 AI의 대표적 활용 사례다. 하지만 유럽 기업 조사에서 예측 정비 구현을 성공적으로 완료한 기업은 **11%** 에 불과하며, 60%는 계획 단계에 머물러 있다([Springer, 2024](https://link.springer.com/article/10.1007/s41471-024-00204-3)).

**예측은 되지만 실행으로 연결되지 않는다 (분석↔실행 단절):**

```
ML 모델이 하는 일:
  "M-14 설비의 48시간 내 고장 확률: 73%"

ML 모델이 하지 않는 일:
  - 고장 확률 73%면 지금 멈춰야 하나, 현재 Lot 완료 후 멈춰야 하나?
  - 정비에 필요한 부품이 재고에 있는가?
  - 정비 인력은 언제 가용한가?
  - 이 설비를 멈추면 생산 계획에 미치는 영향은?
  - 대체 설비로 전환할 수 있는가?
  → 이 모든 판단은 다시 사람의 몫이 된다.
```

ML 모델은 **하나의 숫자(예측값)** 를 제공하지만, 의사결정에는 **여러 객체 간의 관계와 제약 조건**을 종합한 판단이 필요하다. 예측 모델과 실행 시스템 사이의 "Last Mile"이 비어 있다.

### 4-3. 데이터 레이크의 한계

데이터 레이크는 "모든 데이터를 한 곳에 모으면 분석이 쉬워질 것"이라는 전제에서 출발한다. 하지만 현실은 다르다.

```
데이터 레이크의 약속:     "데이터를 한 곳에 모으면 인사이트가 나온다"
데이터 레이크의 현실:     "데이터를 한 곳에 모았더니 늪(Data Swamp)이 됐다"
```

**데이터는 모였지만 의미와 관계가 없다:**
- MES 테이블의 `machine_id` 컬럼과 CMMS 테이블의 `asset_code` 컬럼이 같은 설비를 가리킨다는 것을 데이터 레이크 자체는 모른다.
- 센서 데이터의 `timestamp`와 MES의 `lot_start_time` / `lot_end_time`을 조인하면 "어떤 Lot 가공 중에 센서 이상이 있었는가"를 알 수 있지만, 이 조인 로직은 분석가의 머릿속에만 존재한다.
- Gartner가 예측한 것처럼, 데이터 거버넌스 없이 모인 데이터는 "context 없는 데이터"이며, context 없는 데이터는 가치가 없다([MSRCosmos](https://www.msrcosmos.com/blog/data-lake-failures-how-to-avoid/)).

### 4-4. RPA의 한계

RPA(Robotic Process Automation)는 반복적 수동 작업을 자동화한다. 제조 현장에서는 데이터 입력, 보고서 생성, 시스템 간 데이터 전달 등에 활용된다.

```
RPA가 할 수 있는 일:
  ✓ MES의 생산 실적을 ERP에 자동 입력
  ✓ 일일 품질 보고서를 자동 생성하여 이메일 발송
  ✓ 설비 알람 발생 시 정해진 담당자에게 알림 전송

RPA가 할 수 없는 일:
  ✗ "이 알람이 중요한 알람인가, 무시해도 되는 알람인가?" 판단
  ✗ "진동 이상의 원인이 레시피 변경인가, 공구 마모인가?" 추론
  ✗ "지금 라인을 멈추는 것이 전체 생산 계획에 미치는 영향은?" 계산
  ✗ 상황에 따라 다른 조치를 선택하는 조건부 판단
```

RPA는 **"단순 자동화"** 이지 **"판단 자동화"** 가 아니다. 미리 정해진 규칙(if-then)을 따를 뿐, 맥락을 이해하고 상황에 맞는 판단을 내리지 못한다. 제조 현장의 문제는 대부분 "정해진 규칙으로 처리할 수 없는 예외 상황"이다.

> **핵심 진단:** BI는 시각화만, ML은 예측만, 데이터 레이크는 저장만, RPA는 반복 실행만 한다. 이 네 가지를 모두 도입해도 **"데이터를 이해하고, 관계를 추론하고, 맥락에 맞는 판단을 내리고, 그것을 실행하는"** 통합된 흐름은 만들어지지 않는다. 각각이 해결하는 영역이 다르고, 그 사이를 연결하는 층이 없기 때문이다.

## 5. 온톨로지 기반 접근이 필요한 이유

### 5-1. 위 4가지 문제를 관통하는 본질적 원인

지금까지 분석한 문제들의 공통 근인(Root Cause)은 하나다.

> **"데이터에 의미(Semantics)와 관계(Relationship)와 행동(Action)이 없다."**

| 문제 | 누락된 것 |
|------|-----------|
| 데이터 사일로 | 시스템 간 **공유된 의미(Shared Semantics)**가 없다. 같은 설비를 다른 이름으로 부르고, 같은 개념을 다른 스키마로 표현한다. |
| 품질 추적 불가 | 데이터 간 **관계(Relationship)**가 시스템적으로 정의되지 않았다. Lot→Machine→Tool→Recipe→Material의 연결이 사람 머릿속에만 있다. |
| 의사결정 병목 | **행동(Action)**이 데이터와 분리되어 있다. 데이터를 보고 판단한 후, 별도의 시스템에서 별도의 절차로 실행해야 한다. |
| 기존 접근 실패 | 위 세 가지를 **통합적으로** 해결하지 못하고, 각각을 부분적으로만 다룬다. |

### 5-2. 팔란티어가 이 문제를 재정의한 방식

팔란티어는 수십 개 고객사 배포 경험에서 동일한 문제 패턴을 마주하고, 문제의 프레임 자체를 바꿨다(pre-research.md 참조).

**기존 프레임:** "데이터를 어떻게 저장/이동/분석할 것인가?"
**팔란티어의 프레임:** "어떤 결정을 어떻게 지원할 것인가?"

이 관점 전환에서 나온 핵심 설계가 **온톨로지(Ontology)** 다. 데이터를 테이블이 아니라 **현실 세계의 객체(Object)** 로 재구성하고, SQL JOIN이 아니라 **업무 의미를 가진 관계(Link)** 로 연결하고, 분석 코드가 아니라 **도메인 판단 로직(Function)** 으로 계산하고, 수동 조작이 아니라 **통제된 행동(Action)** 으로 실행한다.

```
기존 데이터 아키텍처:
  테이블 → ETL → 데이터 레이크 → BI 대시보드 → 사람이 보고 판단 → 수동 실행

온톨로지 기반 아키텍처:
  원천 데이터 → 객체(Object) 정의 → 관계(Link) 정의 → 로직(Function) 작성
  → 사용자/AI가 객체 세계에서 탐색 → 판단 → Action 실행 → 결과 반영 (닫힌 루프)
```

### 5-3. 온톨로지가 각 문제를 해결하는 메커니즘

#### 데이터 사일로 → 통합 객체 세계 (Unified Object World)

```
Before:
  ERP의 "EQ-001-A"  ≠  MES의 "MACHINE_A_LINE1"  ≠  CMMS의 "ASSET-00142"
  → 같은 설비, 세 개의 이름, 연결 불가

After (Ontology):
  Object Type: Machine
  ├── machineId: "M-14" (Golden Record, 단일 식별자)
  ├── Backing Datasource:
  │   ├── ERP: equipment.eq_code = "EQ-001-A"
  │   ├── MES: work_station.ws_id = "MACHINE_A_LINE1"
  │   └── CMMS: asset.asset_code = "ASSET-00142"
  ├── Properties: (세 시스템의 속성이 하나의 객체로 통합)
  │   ├── status (MES 실시간)
  │   ├── lastMaintenanceDate (CMMS)
  │   ├── purchaseCost (ERP)
  │   └── currentVibration (SCADA 시계열)
  └── → 어떤 시스템에서 왔든 "Machine M-14"로 접근 가능
```

핵심은 데이터를 이동하는 것이 아니라, **데이터 위에 의미 계층(Semantic Layer)을 올리는 것**이다. 원천 시스템은 그대로 두고, 온톨로지가 "이것은 같은 것이다"를 정의한다.

#### 품질 추적 불가 → 관계 그래프 탐색 (Relationship Graph Traversal)

```
Before:
  Lot, Machine, Tool, Recipe, Material, Operator가 각각의 시스템에 존재
  → 연결하려면 수동으로 4~5개 시스템 데이터를 엑셀에 병합

After (Ontology):
  Lot L-2024-0317
   ├── [processedOn] → Machine M-14
   │                      ├── [mountedWith] → Tool T-33 (누적 847시간)
   │                      └── [usedRecipe] → Recipe R-42 (변경: 3/15)
   ├── [usedMaterial] → Material Lot M-2024-0892
   │                      └── [provides] ← Supplier S-07
   └── [processedOn] → Machine → Shift ← [assignedTo] ← Operator OP-Kim (야간조)

  → 불량 발생 시 이 그래프를 따라가면 수 초 내에 전체 컨텍스트 확보
  → "동일 Recipe를 사용한 다른 Lot에도 불량이 있는가?" 같은 질문도 즉시 답 가능
```

수작업 2~3일이 걸리던 추적이, 관계 그래프 탐색으로 **수 초** 안에 가능해진다.

#### 의사결정 병목 → Action 자동화 (Automated Decision-Action Loop)

```
Before:
  이상 감지 → (전화) → 정보 수집 → (엑셀) → 분석 → (회의) → 결정 → (수동) → 실행
  총 소요: 2시간 ~ 3일

After (Ontology + AI):
  이상 감지 → 시스템이 자동으로 관련 객체/관계를 조합하여 컨텍스트 생성
           → Function이 원인 확률 계산
           → AI가 조치 옵션 제안 (근거와 함께)
           → Action으로 실행 (승인 워크플로 포함)
           → 실행 결과가 다시 객체 상태에 반영 (닫힌 루프)
  총 소요: 수 분
```

팔란티어의 Action Type은 단순 실행이 아니다. **유효성 검사 → 권한 확인 → 승인 워크플로 → 실행 → 감사 로그**가 내장되어 있어, "AI가 멋대로 라인을 멈추는" 일이 구조적으로 불가능하다. 동시에 "사람이 4개 시스템에 로그인해서 수동으로 조치하는" 비효율도 제거한다.

### 5-4. 왜 온톨로지여야 하는가 — 대안 대비 차별점

```
데이터 통합만 하면?     → 데이터 레이크 (의미 없는 저장소, 80% 실패)
관계만 정의하면?        → 그래프 DB (읽기만 가능, 실행 불가)
AI만 올리면?            → LLM 챗봇 (맥락 없는 답변, 할루시네이션)
자동화만 하면?          → RPA (판단 없는 반복 실행)

온톨로지 = 의미(Object) + 관계(Link) + 판단(Function) + 실행(Action) + 보안(Security)
         → 데이터에 "세계관"을 부여하고, 그 세계관 위에서 AI와 사람이 함께 운영
```

팔란티어의 핵심 통찰은 이것이다:

> **데이터를 더 많이 모으거나, 더 빠르게 처리하거나, 더 예쁘게 보여주는 것이 아니라,
> 데이터를 현실 세계의 객체와 관계로 재구성하고, 그 위에 판단과 실행을 얹는 것.
> 이것이 "데이터 중심(Data-Centric)"에서 "결정 중심(Decision-Centric)"으로의 전환이다.**
